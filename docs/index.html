Key observations
  • Model size Dense models above ~16 GB overflow a single 16 GB GPU into system RAM, dropping decode to 4–10 tok/s (DDR5 ~90 GB/s vs GPU 672 GB/s). Dual GPU eliminates the spill: Gemma 31B goes from 4.4 to 29.2 tok/s — a 6.6× lift. MoE models are an exception: despite large GGUF files, only ~10–15% of weights are read per token (active experts), so they reach 60–67 tok/s even on a single GPU.
  • PCIe lanes GPU0 (PCIe 4.0 x16 from CPU, 32 GB/s) outperforms GPU1 (PCIe 4.0 x2 from chipset, 4 GB/s) by 1–8% in decode for VRAM-resident models. For spilling models, both GPUs are identical despite the 8× PCIe bandwidth gap — the bottleneck is CPU-side computation on off-GPU layers, not PCIe transfer. TTFT is slower on GPU1 for large prompts due to chipset pipeline latency.
  • Backend For small dense models, llama.cpp single GPU is 2–5% faster in decode. For MoE on a single GPU, Ollama wins by 25–45% (Qwen 35B MoE: 94.5 vs 67.0 tok/s) via cuBLAS tensor cores tuned for sparse routing. For 27B+ dense models, llama.cpp dual tensor surpasses Ollama by up to 67%. Ollama cannot split one inference across two GPUs.
  • Dual GPU Small models that fit in one GPU: dual provides no benefit (1–3% penalty). Large dense models: dual is transformative — 2–7× by keeping the model in VRAM. MoE: a moderate 35–55% gain. The second GPU is mandatory for 27B+ dense models at usable speeds on this hardware.
  • Tensor split --split-mode tensor runs both GPUs in parallel per layer, doubling effective bandwidth (2 × 672 = 1 344 GB/s). Dense models: 50–70% speedup over layer split. MoE: 12–16% — sparse activation limits per-token reads so allreduce overhead is proportionally larger. The PCIe x2 slot is not a bottleneck for decode: allreduce data is only ~14 KB per layer per token.
